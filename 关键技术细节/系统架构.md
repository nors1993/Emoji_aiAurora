# 架构
aiAurora 采用经典的三层前端架构：

```
表现层（Presentation Layer）4w
  ├── App.tsx（布局 + Error Boundary + 可调节分屏）
  ├── AvatarCanvas.tsx（Three.js 3D 渲染）
  ├── Chat.tsx（聊天 + 语音 + 搜索 + Markdown）
  └── Settings.tsx（配置面板）

状态管理层（State Management）
  └── Zustand chatStore（消息/情绪/设置/人格）

服务层（Service Layer）
  ├── llm.ts（OpenAI SSE + Ollama NDJSON 流式）
  ├── asrClient.ts（FunASR WebSocket 客户端）
  ├── ttsClient.ts（Qwen3-TTS SSE 流式客户端）
  ├── audioCapture.ts（麦克风 PCM 采集）
  ├── streamParser.ts（SSE 流式解析器）
  └── webSearch.ts（多 Provider 搜索链）
```

**设计理由**：

1. **关注点分离**：UI 渲染、业务状态、通信逻辑明确分层，每个文件职责单一。
2. **可测试性**：服务层（llm/asr/tts）可脱离 UI 独立测试。
3. **可替换性**：如从 OpenAI 切换到 Ollama，只需修改 llm.ts，不影响 UI。
4. **Electron 适配**：渲染进程与主进程通过 `contextBridge` 隔离，符合 Electron 安全最佳实践。

# 为什么选择 Zustand 而不是 Redux 或 Context API 做状态管理？
选择 Zustand 的原因：

| 维度       | Zustand                      | Redux Toolkit                  | Context API            |
| ---------- | ---------------------------- | ------------------------------ | ---------------------- |
| 样板代码   | 极少（~10 行即可创建 store） | 较多（slice/reducer/dispatch） | 最少但需 Provider 嵌套 |
| 性能       | ✅ 精确订阅，避免不必要渲染   | ✅ 通过 selector 控制           | ❌ 所有消费者重新渲染   |
| 体积       | ~1KB                         | ~12KB                          | 内置                   |
| 持久化     | 手动 localStorage            | 需 redux-persist               | 手动                   |
| TypeScript | 天然支持                     | 支持但类型较复杂               | 需额外泛型             |
| 异步       | 内置支持                     | 需 createAsyncThunk            | 需自定义               |

具体到本项目：

- **轻量化需求**：项目不需要 Redux 的中间件生态（如 saga/thunk），消息、情绪、设置三个核心状态用 Zustand 完全够用。
- **性能敏感**：Chat 组件和 AvatarCanvas 需要频繁读取状态，Zustand 的精确订阅避免了两者互相影响。
- **初始化简洁**：`loadSettingsFromStorage()` 在 store 创建时同步读取 localStorage，确保首次渲染即使用用户配置，避免了 Redux 那种 store 初始化后还要 dispatch 的异步过程。

# 如果用户量从单人扩展到 10 万 DAU，项目的架构会遇到哪些瓶颈？如何改进？
**当前架构局限**：

- 所有 AI 推理在本地或单用户 API 进行，无服务端聚合。
- 状态存储仅 localStorage，无云端同步。

**瓶颈分析与改进方案**：

| 瓶颈                 | 分析                                 | 改进方案                                                     |
| -------------------- | ------------------------------------ | ------------------------------------------------------------ |
| **LLM API 成本**     | 每个用户独立调用 OpenAI API          | 引入请求缓存（相同 query 缓存响应）、promot 优化减少 token 消耗 |
| **模型推理负载**     | ASR/TTS 部署在本地 GPU，单机无法服务 | 转为云端推理集群，使用任务队列 + GPU 弹性伸缩                |
| **无用户系统**       | localStorage 无法跨设备同步          | 添加后端认证（JWT）+ 云端持久化（MongoDB/PostgreSQL）        |
| **WebSocket 连接数** | 每个 ASR 连接独占一条 WebSocket      | 使用连接池 + 消息队列，拆分为"音频上传"和"结果推送"两个通道  |
| **3D 渲染性能**      | 移动端/低端设备 Three.js 可能卡顿    | 添加 WebGL 降级策略：检测 GPU 能力，自动降低面数/关闭特效    |
| **无 CDN**           | Vite build 后直接 serve，无缓存优化  | 静态资源部署 CDN（Cloudflare/阿里云），设置合理 Cache-Control |
| **无 APM**           | 无法监控用户体验                     | 接入 Sentry/RUM 监控，追踪 JS 错误、API 延迟、用户操作漏斗   |
